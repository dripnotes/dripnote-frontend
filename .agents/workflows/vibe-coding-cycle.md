# 바이브 코딩 개발 파이프라인 (Vibe Coding Cycle)

이 프로젝트는 하네스 엔지니어링(Harness Engineering) 환경을 적용하여 AI 에이전트와 작업할 때, 일관성 유지와 안정적인 코드 품질을 위해 반드시 다음의 **6단계 주기(Cycle)**를 순차적으로 따릅니다.

## 1. 계획 (Planning)

- 사용자가 `specs/*.md`에 요구사항을 작성하거나 에이전트에게 개발할 기능을 제시합니다.
- **Action:** AI 에이전트는 코드를 즉시 수정하지 않고, 요구사항과 현재 아키텍처를 분석합니다. 이때 반드시 **`specs/common-ui-spec.md`를 먼저 확인하여 디자인 및 설계의 일관성을 확보**한 뒤, 프로젝트 최상단 루트의 `PLAN.md` 파일에 구체적인 **Implementation Plan (구현 계획안)**을 작성(업데이트)합니다.

## 2. 허가 (Approval)

- AI 에이전트가 제안한 계획안을 기반으로 사용자에게 보고한 뒤 피드백을 대기합니다.
- **Action:** 사용자는 제안된 구조와 로직이 요구사항 명세서와 일치하는지 리뷰하고, 구두 승인("진행해", "좋아" 등)을 내립니다. 승인 전까지 코드는 변경되지 않습니다.

## 3. 구현 (Implementation)

- 허가가 떨어지면 수립된 계획과 태스크(Task)에 맞춰 차례대로 코드를 작성합니다.
- **Action:** 에이전트는 `ui-library`, 전역 린트 설정 등 모노레포의 규약을 준수하며 코드를 구현합니다. 진행도는 체크리스트를 통해 공유됩니다.

## 4. 리뷰 및 테스트 (Review & Test)

- 1차적인 코드 작업이 끝나면, 작성된 코드의 문법, 타입 체크, 린트 및 터미널 명령어(빌드/테스트)를 통해 검증합니다.
- **Action:** 문제가 있다면 이 단계에서 에이전트가 스스로 에러 코드를 확인하고 수정(Self-Correction)합니다.

## 5. 최종 확인 (Final Verification)

- 테스트를 통과한 결과물을 사용자가 직접 눈으로 보고(`pnpm dev`) 조작해 봅니다.
- **Action:** UI의 미적인 부분이나 상세 비즈니스 로직에 대해 피드백을 받고 폴리싱(Polishing) 작업을 수행합니다.

## 6. 문서 업데이트 (Documentation Update)

- 기능 구현이 완벽히 끝나면, 개발된 내용을 바탕으로 명세서를 최신화합니다.
- **Action:** 새로 제정된 컨벤션이 있다면 `.agents/rules/`에 반영하고, 변경 내역을 `walkthrough.md` 또는 커밋 메시지에 남겨 다음 개발(Cycle)을 준비합니다.
