# CHG-260506-01: 소셜 로그인 아키텍처 개편 (BFF & 2단 리다이렉트)

## 1. 목적 (Purpose)

- 실제 백엔드 API 주소를 노출하지 않기 위해 BFF(Backend For Frontend) 구조를 도입합니다.
- 로그인 성공 후 사용자가 원래 머물던 페이지로 정확히 복구하기 위해 `redirect_to` 쿠키 기반의 2단 리다이렉트 시스템을 구축합니다.

## 2. 과정 및 변경 사항 (Process & Deviations)

### 2.1 BFF 라우트 핸들러 구현

- `apps/web/app/api/auth/[provider]/route.ts` 신규 생성: 외부 로그인 리다이렉트 처리 및 보안 쿠키 포워딩.
- `apps/web/app/api/auth/refresh/route.ts` 신규 생성: 클라이언트로부터 받은 Refresh Token을 백엔드에 전달하여 Access Token으로 교환하는 프록시 역할 수행.
- 실제 백엔드 주소는 `apps/web/.env.local`의 `BACKEND_URL` 환경변수를 통해 관리하며 외부에 노출되지 않도록 설계했습니다.

### 2.2 클라이언트 사이드 흐름 제어 및 목적지 저장

- `SocialButton.tsx` 수정: 클릭 시 현재 경로를 `redirect_to` 쿠키에 저장 후 BFF(`/api/auth/[provider]`) 호출.
- **보안**: 저장 전 해당 경로가 내부 경로(`isValidInternalPath`)인지 검증하여 오픈 리다이렉트 취약점을 예방했습니다.

### 2.3 미들웨어 기반 환승역(Success Station) 처리

- `apps/web/app/(protected)/auth/success/page.tsx` 신규 생성 (환승역).
- `middleware.ts` 구현: 해당 경로 요청을 서버 사이드에서 가로채어 BFF(`/api/auth/refresh`)와 통신하고, 발급된 토큰을 쿠키에 설정한 후 최종 목적지로 302 리다이렉트 시킵니다.
- **오픈 리다이렉트 방지**: 미들웨어에서 목적지 쿠키를 읽을 때 내부 경로 여부를 다시 한번 검증하고 유효하지 않을 경우 `/`로 강제 고정합니다.

### 2.4 기존 유산 정리

- 더 이상 사용되지 않는 `apps/web/app/login/callback` 및 `apps/web/app/auth/success/page.tsx` (루트) 경로를 완전히 제거하고 `(protected)` 그룹으로 일원화했습니다.

## 3. 결과 (Result)

- 보안성 향상: 브라우저 네트워크 탭에서 실제 백엔드 주소가 노출되지 않으며, 토큰이 JavaScript에서 접근 불가능한 `HttpOnly` 쿠키로 관리됩니다.
- UX 향상: 로그인 후 사용자가 보던 페이지로 끊김 없이 즉시 돌아올 수 있습니다.

## 4. 검증 및 지표 (Verification & Metrics)

- `redirect_to` 쿠키를 통한 목적지 주소 보존 및 미들웨어 리다이렉트 동작 확인 완료.
- BFF를 통한 외부 로그인 페이지 리다이렉트 및 보안 쿠키(`getSetCookie`) 포워딩 정상 동작 확인.
- 내부 경로 검증 로직(`isValidInternalPath`)을 통해 비정상적 리다이렉트 차단 확인.
