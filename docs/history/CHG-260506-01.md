# CHG-260506-01: 소셜 로그인 아키텍처 개편 (BFF & 2단 리다이렉트)

## 1. 목적 (Purpose)

- 실제 백엔드 API 주소를 노출하지 않기 위해 BFF(Backend For Frontend) 구조를 도입합니다.
- 로그인 성공 후 사용자가 원래 머물던 페이지로 정확히 복구하기 위해 `sessionStorage` 기반의 2단 리다이렉트 시스템을 구축합니다.

## 2. 과정 및 변경 사항 (Process & Deviations)

### 2.1 BFF 라우트 핸들러 구현

- `apps/web/app/api/auth/[provider]/route.ts` 신규 생성: 외부 로그인 리다이렉트 처리.
- `apps/web/app/api/auth/refresh/route.ts` 신규 생성: 클라이언트로부터 받은 Refresh Token을 백엔드에 전달하여 Access Token으로 교환하는 프록시 역할 수행.
- 실제 백엔드 주소는 `apps/web/.env.local`의 `BACKEND_URL` 환경변수를 통해 관리하며 외부에 노출되지 않도록 설계했습니다.

### 2.2 클라이언트 사이드 흐름 제어 및 토큰 교환

- `SocialButton.tsx` 수정: 클릭 시 현재 경로를 `sessionStorage`에 저장 후 BFF(`/api/auth/[provider]`) 호출.
- `apps/web/app/auth/success/page.tsx` 수정: URL 파라미터 대신 **HttpOnly 쿠키**에 담긴 `refreshToken`을 사용하여 BFF(`/api/auth/refresh`)를 통해 Access Token으로 교환하도록 설계했습니다.
- **백엔드 명세 준수**: 토큰 재발급 요청 시 `Refresh-Token` 헤더(접두사 없음)를 사용하고, `data.accessToken` 구조의 JSON 응답을 처리하도록 구현했습니다.
- **보안**: 오픈 리다이렉트 방지 로직과 `HttpOnly` 쿠키 활용을 통해 토큰 탈취 위험을 최소화하고 백엔드 엔드포인트를 완전히 은닉했습니다.

### 2.3 환승역(Success Station) 구현

- `apps/web/app/auth/success/page.tsx` 신규 생성.
- 백엔드로부터 토큰을 전달받아 저장(`authUtils`)하고, `sessionStorage`에서 목적지를 읽어 `router.replace`를 통해 최종 이동시킵니다.
- **보안**: 오픈 리다이렉트 공격을 방지하기 위해 타겟 주소가 동일 도메인 내부 경로(`/`로 시작)인지 검증하는 로직을 추가했습니다.

### 2.4 기존 유산 정리

- 더 이상 사용되지 않는 `apps/web/app/(auth)/login/callback` 경로와 코드를 완전히 제거했습니다.

## 3. 결과 (Result)

- 보안성 향상: 브라우저 네트워크 탭에서 실제 백엔드 주소가 노출되지 않습니다.
- UX 향상: 로그인 후 사용자가 보던 페이지(예: 원두 상세 정보, 놀이터 등)로 즉시 돌아올 수 있습니다.

## 4. 검증 및 지표 (Verification & Metrics)

- `sessionStorage`를 통한 목적지 주소 보존 및 복구 동작 확인 완료.
- BFF를 통한 외부 로그인 페이지 리다이렉트 정상 동작 확인.
- 내부 경로 검증 로직을 통해 외부 도메인으로의 비정상적 리다이렉트 차단 확인.
